2) Request Body (JSON)
{

  "text": "⦃v:1⦄Hello ⦃r:soft⦄world⦃w:250⦄!"		// required: UTF-8, includes tokens v/r/w/c only
}

3) Token Semantics (in text)
- ⦃v:n⦄			Voice switch. Ends current segment and starts a new one; also **clears** recipe stack.
- ⦃r:name⦄		Push recipe 'name' (stacked). Ends current segment; new one starts if there is speech.
- ⦃r⦄			Pop (unstack) one recipe. Ends current segment; new one starts if there is speech.
- ⦃c:name⦄		Channel: left, right, center, in-head. handled by SoX
- ⦃w:ms⦄			Insert an inline silence of ms milliseconds at that exact point (non-breaking). 
				(values: integer ≥0 and ≤ WAIT_MAX_MS=10000)

Notes:
• Only ⦃v⦄, ⦃r⦄/⦃r:name⦄, ⦃c:name⦄ and ⦃w:ms⦄ are recognized. No aside markers; client handles any aside logic.
• Segments are formed by v/r/c boundaries; order is preserved; empty segments are not allowed out but may be presented to server. 

4) Effective Parameter Resolution (per speech segment)
Let the current voice DB row provide baselines (s_base, speed_base, pitch_base, n_base, w_base, x_base, vol_base) from their existing field names.
Let the active recipe stack (top-most last) provide optional multipliers/overrides.

Compute:
- speed_eff = round( speed_base × Π(recipe.mul_speed or 1.0), 2 )	// translated to "length" for Piper
- pitch_eff = round( pitch_base × Π(recipe.mul_pitch or 1.0), 2 )		// handled by SoX
- n_eff = round( n_base × Π(recipe.mul_n or 1.0), 2 )
- w_eff = round( w_base × Π(recipe.mul_w or 1.0), 2 )
- x_eff = (top-most recipe.abs_x if any, else x_base)
- volume_eff = vol_base × Π(recipe.vol_mul or 1.0), 2)			// handled by Piper

All “mul_*” are multiplicative
Volume handled by Piper, not Sox
Stereo, channel, and pitch handled by Sox
